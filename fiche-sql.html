Tables de référence

users, roles, notes — structures utilisées dans les exemples

users(id, nom, prenom)

Ex: (1, Dupont, Jean), (2, Martin, Marie), (3, Dupont, Paul)

roles(nom, role)

Ex: (Dupont, admin), (Martin, user)

notes(nom, valeur)

Ex: (Alice, 12), (Alice, 14), (Bob, 10) — notes des étudiants

GROUP BY

GAG — + colonnes = + lignes

Règle GAG

Chaque colonne du SELECT doit être soit dans le GROUP BY, soit dans une fonction d'agrégat (COUNT, SUM, AVG, MAX, MIN).

SELECT nom, prenom, count(*) FROM users GROUP BY nom
→ prenom ni dans GROUP BY ni agrégat = invalide
SELECT nom, count(*) FROM users GROUP BY nom, prenom
→ valide

Nombre de lignes

GROUP BY nom, prenom crée plus de groupes que GROUP BY nom seul → plus de lignes en résultat.

Ex: users a 3 lignes (Dupont/Jean, Dupont/Paul, Martin/Marie). GROUP BY nom → 2 lignes. GROUP BY nom, prenom → 3 lignes.

Mémo : Plus on divise les groupes, plus on a de lignes.

Questions type

Q1
R1: SELECT nom FROM users GROUP BY nom — R2: SELECT nom FROM users GROUP BY nom, prenom. Laquelle donne le plus de lignes ?
→ Voir la réponse
R2. Plus de colonnes dans GROUP BY = plus de groupes = plus de lignes.
Q2
SELECT nom, prenom, count(*) FROM users GROUP BY nom — valide ou invalide ?
→ Voir la réponse
Invalide. prenom est dans SELECT mais ni dans GROUP BY ni dans un agrégat (règle GAG).

Agrégats imbriqués

Pas de poupée russe

On ne peut pas imbriquer des agrégats directement. Un seul niveau à la fois.

SELECT max(count(*)) FROM notes GROUP BY nom
→ invalide

Pour faire équivalent : utiliser une sous-requête.

SELECT max(c) FROM (SELECT count(*) as c FROM notes GROUP BY nom)
Mémo : Pas de max(count()), sum(avg()), etc. directement.

Questions type

Q
SELECT nom, max(count(*)) FROM notes GROUP BY nom — que renvoie cette requête ?
→ Voir la réponse
Rien, elle est invalide. On ne peut pas imbriquer max(count()) directement.
Q
SELECT nom, sum(valeur)/count(*) FROM notes GROUP BY nom — valide ?
→ Voir la réponse
Valide. sum() et count() sont au même niveau (pas imbriqués), pas de max(count()).

DELETE / TRUNCATE / DROP

Données → Vide → Détruit

DELETE FROM table

Supprime les lignes. La table reste, structure intacte. Peut avoir WHERE.

TRUNCATE TABLE table

Vide la table (toutes les lignes). La structure reste. Plus rapide que DELETE.

DROP TABLE table

Supprime la table entière (structure + données). La table n'existe plus.

Mémo : DELETE = données, TRUNCATE = vide, DROP = destruction totale.

Questions type

Q
DELETE FROM users; — que fait cette requête ?
→ Voir la réponse
Efface les données mais pas la table. La structure (colonnes, etc.) reste. La table existe toujours, vide.
Q
TRUNCATE TABLE users; — la table existe encore après ?
→ Voir la réponse
Oui. TRUNCATE vide la table, la structure reste. Pour supprimer la table : DROP TABLE.

IN (sous-requête)

IN = 1 colonne

La sous-requête après IN doit renvoyer exactement une colonne. Pas deux, pas zéro.

WHERE nom IN (SELECT nom, prenom FROM users)
→ 2 colonnes = invalide
WHERE nom IN (SELECT nom FROM users)
→ 1 colonne = valide
Mémo : IN compare une valeur à une liste → la liste = 1 colonne.

Questions type

Q
SELECT * FROM roles WHERE nom IN (SELECT nom, prenom FROM users) — valide ?
→ Voir la réponse
Non valide. La sous-requête renvoie 2 colonnes (nom, prenom), IN exige 1 seule colonne.
Q
SELECT * FROM roles WHERE nom IN (SELECT nom FROM users) — valide ?
→ Voir la réponse
Valide. 1 colonne dans la sous-requête. Renvoie les rôles dont le nom est dans users.

EXISTS / NOT EXISTS

Test de présence → 0 ou des lignes

EXISTS

Retourne TRUE si la sous-requête renvoie au moins une ligne. Le contenu des colonnes n'a pas d'importance (souvent SELECT 1).

NOT EXISTS

Retourne TRUE si la sous-requête ne renvoie aucune ligne.

Piège classique

Si la sous-requête renvoie toujours des lignes (ex: SELECT nom FROM users sans WHERE), alors NOT EXISTS est toujours FALSE → 0 résultat.

SELECT * FROM users WHERE NOT EXISTS (SELECT nom FROM users)
→ sous-requête lit toute la table → toujours des lignes → NOT EXISTS = faux → 0 ligne
Mémo : NOT EXISTS + sous-requête qui renvoie tout = jamais de résultat.

Questions type

Q
SELECT * FROM users WHERE NOT EXISTS (SELECT nom FROM users) — des résultats ?
→ Voir la réponse
Non, jamais. La sous-requête renvoie toujours des lignes (toute la table). NOT EXISTS = toujours faux → 0 ligne.
Q
SELECT * FROM users u WHERE EXISTS (SELECT 1 FROM roles r WHERE r.nom = u.nom) — valide ?
→ Voir la réponse
Valide. Sous-requête corrélée (u.nom). Renvoie les users qui ont un rôle correspondant.

Corrélée vs non corrélée

Préfère non corrélée

Sous-requête non corrélée

Indépendante de la requête externe. Exécutée une seule fois. Plus rapide.

SELECT * FROM users WHERE nom IN (SELECT nom FROM roles)

Sous-requête corrélée

Référence une colonne de la requête externe (ex: u.nom). Exécutée pour chaque ligne. Plus lente.

SELECT * FROM users u WHERE EXISTS (SELECT 1 FROM roles r WHERE r.nom = u.nom)
Mémo : Si les deux solutions marchent, choisis la non corrélée (performance).

Questions type

Q
Deux solutions : une avec sous-requête corrélée, l'autre non corrélée. Laquelle préférer ?
→ Voir la réponse
Non corrélée. Exécutée 1 fois vs N fois (pour chaque ligne). Meilleure performance.

JOINs

INNER / LEFT / RIGHT — LEFT/RIGHT ≥ INNER

INNER JOIN

Seulement les lignes qui matchent des deux côtés. Intersection.

LEFT JOIN

Toutes les lignes de la table de gauche + les matchs à droite. Pas de match = NULL à droite.

RIGHT JOIN

Toutes les lignes de la table de droite + les matchs à gauche.

Ordre des résultats

LEFT JOIN garde tout à gauche → toujours ≥ INNER. RIGHT JOIN garde tout à droite → toujours ≥ INNER. Le résultat de INNER est inclus dans LEFT et RIGHT.

R1: INNER JOIN → seulement les matchs
R2: LEFT JOIN → tous les users + matchs
→ R2 a toujours ≥ lignes que R1
Mémo : LEFT = tout à gauche, RIGHT = tout à droite, INNER = matchs uniquement.

Exemple avec données

users: (1,Dupont,Jean), (2,Martin,Marie) — roles: (Dupont,admin)

RequêteRésultat
INNER JOIN1 ligne (Dupont)
LEFT JOIN users2 lignes (Dupont+admin, Martin+NULL)

Questions type

Q
R1: INNER JOIN — R2: LEFT JOIN (table gauche = users). Qui donne le plus de lignes ?
→ Voir la réponse
R2 (LEFT JOIN). Garde tous les users. INNER ne garde que les matchs. R2 ≥ R1 toujours.
Q
Si tous les users ont un rôle, INNER et LEFT donnent le même nombre de lignes ?
→ Voir la réponse
Oui. Si tout matche, pas de NULL → même résultat. Mais en général LEFT ≥ INNER.
